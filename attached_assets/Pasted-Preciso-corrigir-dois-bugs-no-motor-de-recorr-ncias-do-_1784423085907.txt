Preciso corrigir dois bugs no motor de recorrências do MyFinance: services/recurringEngine.ts e pages/Scheduled.tsx. Este último já foi atualizado recentemente (agora usa TransactionFormDialog em vez do antigo ScheduledFormDialog, e tem seleção múltipla com BulkEditScheduledDialog) — não desfaça essas mudanças, só ajuste a lógica de geração/regeneração de pendentes.

1. IMPEDIR DUPLICAÇÃO AO "REGENERAR" UMA RECORRÊNCIA JÁ PAGA
- Em services/recurringEngine.ts, a função regeneratePendingForScheduled (linha ~199) hoje verifica apenas se já existe uma transação com status = 'pending' para aquele scheduled_id + reference_month antes de inserir uma nova. Troque essa verificação para checar QUALQUER transação (pending OU paid) vinculada a scheduled_id + reference_month.
- Se já existir uma transação PAGA para esse período: não crie nada. Mude o retorno da função para algo que distinga esse caso do "gerou com sucesso", por exemplo trocando o tipo de retorno de `Promise<Transaction | null>` para `Promise<{ transaction: Transaction | null; reason?: 'already_paid' | 'already_pending' }>`.
- Atualize a chamada em context/FinanceContext.tsx (regeneratePendingTransaction, que hoje chama regeneratePendingForScheduled) para repassar esse novo formato de retorno.
- Em pages/Scheduled.tsx, dentro de handleRegenerate (linha ~88), trate o novo retorno: se reason === 'already_paid', mostre toast.info('Este período já foi pago — nada para regenerar.') em vez de criar duplicata. Mantenha os outros casos (pending existente, sucesso) como estão hoje.

2. GARANTIR QUE CONTAS ATRASADAS (NÃO PAGAS DE MESES ANTERIORES) CONTINUEM VISÍVEIS E ACUMULANDO
- O motor de recorrência só considera o MÊS ATUAL (getExpectedReferenceMonth sempre retorna o mês de agora). Pendências não pagas de meses anteriores continuam existindo em transactions, mas o resumo "Mês atual" da tela de Recorrentes (currentPeriodStatus, linha ~54 de pages/Scheduled.tsx) só olha pro mês corrente, então elas somem da visão geral mesmo continuando pendentes.
- Crie uma nova função em services/recurringEngine.ts, ex: getOverdueTransactions(scheduledList: ScheduledTransaction[], transactions: Transaction[]): Transaction[], que filtra transações com status 'pending', scheduledId preenchido, vinculadas a uma recorrência ativa, e date < hoje (string 'YYYY-MM-DD' comparada com getTodayStr()) — ordenadas da mais antiga para a mais nova. Não precisa acessar o banco, pode operar só sobre os arrays já carregados no FinanceContext (scheduled e transactions).
- Em pages/Scheduled.tsx, use essa função (via useMemo) para calcular a lista de atrasadas, e adicione uma seção de alerta ACIMA da tabela principal (abaixo do cabeçalho "Recorrentes"), visível apenas quando houver 1+ itens atrasados: título "Você tem N conta(s) atrasada(s) somando R$ X,XX", listando cada uma (descrição, categoria, data de vencimento formatada com formatShortDate, valor) com um botão "Marcar como paga" que chama updateTransaction(tx.id, { status: 'paid' }) direto dali (updateTransaction já vem de useFinance()).
- Na coluna "Mês atual" da tabela principal (componente PeriodBadge, linha ~120), quando o status for 'pending' E a transação correspondente tiver date < hoje, mostre uma badge "Atrasado" em vermelho (ex: variant destructive, ícone AlertCircle) em vez da badge âmbar "Pendente" — isso exige buscar a transação correspondente (não só o status) dentro de currentPeriodStatus ou de um novo map equivalente, para acessar a data dela.
- generateAllPendingForCurrentPeriod (chamada automaticamente pelo FinanceContext) deve continuar gerando a pendência do mês atual normalmente mesmo havendo pendências atrasadas de meses anteriores — as duas devem coexistir sem conflito (isso já deveria funcionar hoje, pois reference_month é diferente entre elas; apenas confirme que não há regressão).
